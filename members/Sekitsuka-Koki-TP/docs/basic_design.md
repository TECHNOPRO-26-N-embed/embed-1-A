## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                 【Arduino UNO R3】            【出力】
膜スイッチ(4x4) ────→┐                         ┌──→ 赤LED（不一致）
                    ├ 4桁入力受付・判定処理     ├──→ 黄LED（位置違い一致）
                       状態管理・試行回数管理   ├──→ 緑LED（完全一致）
                      └─────────────────────────┘
                                               └──→ （任意）ブザー
```

補足説明（図で表現しきれない場合）：
- 文字ディスプレイは使わず、判定結果はLEDの色で表示する。
- 判定開始後は、入力した4桁の各桁について「左から順に1桁ずつ」結果を出す。
- 判定開始から各桁の反応開始は1.0秒以内とし、1桁あたり0.2秒点灯（順次表示）とする。
- 赤=不一致、黄=数字一致（位置違い）、緑=完全一致を示す。
- `*`は判定開始、`#`は1文字削除として扱う。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（正解コード生成完了）
[入力待ち] ──（4桁入力後に`*`押下）──→ [判定表示] ──（4桁すべて緑）──→ [クリア]
    │                 │                                 │
    │                 └─（4桁未満で`*`）──→ [ミス入力通知] ──（通知後）──→ [入力待ち]
    └─（判定後/`#`で修正）──────────────────────────────────────┘
                      └─（不正解かつ試行回数<10）──────────────→ [入力待ち]
                      └─（不正解かつ試行回数=10）──────────────→ [入力待ち]（ゲーム再初期化）
                                                                               └─（`*`押下で新規開始）──→ [入力待ち]


```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 入力待ち | 膜スイッチ入力を受け取り、4桁バッファを更新する（`#`で1文字削除） | 通常消灯（入力確定時に短く緑点灯） |
| 判定表示 | 入力4桁と正解4桁を比較し、1桁ずつ結果を表示する。不正解時は試行回数を加算し、10回到達時はゲームを再初期化して入力待ちへ戻す | 緑/黄/赤LEDを1桁あたり0.2秒で順次点灯。不正解時は赤LEDを短く点滅させて失敗演出を行う |
| クリア | 正解時の終了状態。次ゲーム開始待ち | 赤・黄・緑LEDを波打つように順次点灯（任意で成功音） |
| ミス入力通知 | 4桁未満で`*`押下などの不正入力を通知する | 赤LEDを0.5秒以内に点滅 |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

## 2. ソフトウェア設計

### 2-1. データ設計（使う変数・定数の洗い出し）

> ※ プログラムで扱う「情報の種類」を先に整理します。
> ソフトウェアワークシート（AI駆動開発）でやった「データの洗い出し」と同じです。

| 情報の名前 | 何を表すか | 型（イメージ） | バイト数 | 初期値 | 備考 |
|:--|:--|:--|:--|:--|:--|
| currentState | 現在の状態（入力待ち/判定表示/クリア/ミス入力通知） | byte | 1B | 0 | enumで管理 |
| secretCode[4] | 正解4桁コード（重複なし） | byte配列 | 4B | {0,0,0,0} | 起動時に生成 |
| inputCode[4] | ユーザー入力4桁 | byte配列 | 4B | {0,0,0,0} | キー入力で更新 |
| judgeResult[4] | 判定結果（2:緑,1:黄,0:赤） | byte配列 | 4B | {0,0,0,0} | 表示順制御に使用 |
| inputLength | 現在の入力桁数 | byte | 1B | 0 | 0〜4 |
| attemptCount | 現在の試行回数 | byte | 1B | 0 | 上限10回 |
| lastKey | 直近で確定したキー値 | char | 1B | '\0' | デバウンス用 |
| lastKeyMillis | 最後にキーを受理した時刻 | unsigned long | 4B | 0 | 30〜50ms判定 |
| lastLedMillis | 最後にLED表示を更新した時刻 | unsigned long | 4B | 0 | 逐次表示タイミング |
| ledIndex | 何桁目を表示中か | byte | 1B | 0 | 0〜3 |

メモ（型選定の理由要約）：
- `byte` は 0〜255 の小さい値（状態番号・桁・回数）に十分で、UNOのSRAM節約に有利。
- `char` は `*` `#` 数字キーなど「1文字の入力値」を扱うために使用。
- `unsigned long` は `millis()` の時刻管理に合わせて使用（オーバーフローまでの時間を確保）。
- `int` は今回の値域では過剰になりやすいため、必要最小限の場面のみ使用する方針。

> [!CAUTION]
> **SRAM使用量チェック（Arduino UNO R3 の上限は 2048B）**
>
> グローバル変数の合計: 約25B
>
> | 合計バイト数 | 判定 |
> |:--|:--|
> | ～1000B | ✅ 余裕あり |
> | 1000～1500B | ⚠️ char配列の長さを見直す |
> | 1500B超 | 🔴 AIに相談し変数を整理すること |

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| 膜スイッチスキャン（`readKeypad()`） | 10ms程度 | **millis** | 入力取りこぼしを防ぎつつ、処理負荷を抑えるため |
| デバウンス判定（`lastKeyMillis`） | 30〜50ms | **millis** | チャタリング除去と応答性（0.2秒以内）の両立のため |
| 判定開始可否チェック（`startJudgeIfReady()`） | キー入力ごと（即時） | **millis** | `*`押下時に即時判定し、未入力時はすぐエラー通知するため |
| 判定結果生成（`judgeAndStoreResult()`） | 判定開始時に1回 | **millis** | 配列計算を短時間で完了し、表示処理と分離するため |
| 判定結果LEDの1桁表示（`updateLedOutput()`） | 0.2秒/桁 | **millis** | 判定開始から各桁の反応開始を1.0秒以内に収めるため |
| 不正入力時の赤点滅（`notifyInvalidInput()`） | 0.5秒以内に通知開始 | **millis** | 非機能要件（0.5秒以内通知）を満たすため |
| 任意: 失敗時赤点滅演出（`playFailBlink()`） | 100ms点灯/100ms消灯を2回 | **millis** | 失敗を明確化しつつ、他処理停止を避けるため |
| 任意: クリア時3色波演出（`playClearWave()`） | 100ms/色で1サイクル | **millis** | クリア演出を行いながら入力監視停止を最小化するため |
| 任意: ブザー通知（`playBuzzerPattern()`） | 判定完了時に1回 | **millis** | LED表示・入力監視と並行動作させるため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。


> [!NOTE]
> **delayを避ける理由**:
> LED表示中に入力が不要な場合でも、将来的な拡張性や他の処理との一貫性を考慮し、`millis`を使用しています。
> `delay`を使うと、他の処理が停止し、機能追加時にコードの修正が難しくなる可能性があります。

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| チャタリング | キー押下直後に接点が揺れる | 前回確定時刻から30〜50ms未満の入力 | デバウンス処理で無視 |
| 入力不足で`*`押下 | 4桁未満で判定開始したとき | `inputLength < 4` を判定 | 判定せず赤LED点滅（0.5秒以内） |
| 範囲外キー入力 | 数字/`*`/`#`以外が読まれたとき | キー値をホワイトリスト照合 | 入力バッファを更新せず無視 |
| 試行回数上限到達 | 10回失敗したとき | `attemptCount >= 10` を判定 | 失敗通知後に新規ゲームへリセット |
| 任意: ジョイスティック揺れ | 閾値近傍で値が変動 | 連続サンプル比較で安定判定 | デッドゾーンを設定して誤選択防止 |

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
- ピン競合は見当たらない（D2〜D12, A0の用途分離）。D0/D1未使用で問題なし。
- 状態遷移は「入力待ち → 判定表示 → クリア/入力待ち」に整理できており、不正入力時の戻り先も定義されている。
- `delay()`多用は入力取りこぼしの原因になるため、キー監視とLED表示は`millis()`で並行処理するのが妥当。

**対応した内容：**
- キー監視、判定表示、不正入力通知を`millis()`ベースで設計した。
- 不正入力（4桁未満`*`）時の専用状態を追加し、入力待ちに戻る遷移を明記した。
- ピン割り当てを表で固定し、グループ統一仕様にRows/Cols順を追記した。

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
- チャタリング対策は必須で、30〜50ms程度のデバウンス時間が妥当。
- `delay()`は判定表示中に入力を止めるため非推奨。LED表示の時間管理も`millis()`で行うべき。
- アナログ入力（任意のジョイスティック）を使う場合はデッドゾーン設定でノイズ対策が必要。

**対応した内容：**
- `lastKeyMillis`を用いたデバウンス仕様を追加した。
- タイミング設計表の主要処理を`millis()`に統一した。
- 任意機能向けにジョイスティックのデッドゾーン方針を異常系に追加した。
